# ADR-0034: Canonical Analytic Programming Integration

## Status

`Accepted`

## Decision Date

`2026-07-11`

## Context

FrameNest previously carried copied Analytic Programming (AP) protocol files in
the repository root:

```text
AP.md
AP_ORCHESTRATOR.md
AP_WORKER.md
BOOT_ORCHESTRATOR.md
BOOT_WORKER.md
NEXT_ORCHESTRATOR.md
NEXT_WORKER.md
```

Those files mixed universal AP rules, FrameNest-specific operating rules,
session handoff state, stale expected commits, and command-observed host facts.
That made the live tree harder to reason about because universal protocol
updates required manual replacement and permanent BOOT/NEXT files encouraged
session state to look like durable project truth.

The canonical AP repository now provides a pinned submodule distribution model,
a dependency-free integration tool, a managed root `AGENTS.md` block, dynamic
restoration prompts, and exceptional repository handoffs only when
unreconstructable state exists.

Canonical AP repository identity:

```text
https://github.com/cisarik/ap.git
github.com/cisarik/ap
```

Initial AP commit:

```text
c4c69f52b9995c609248cee5d04223dbddd6da5f
```

Initial AP subject:

```text
fix: harden universal AP integration workflow
```

## Decision

FrameNest adopts the canonical AP repository as a pinned Git submodule at:

```text
.ap/
```

The consuming project records the exact AP version through the `.ap` gitlink in
the FrameNest commit. Universal AP protocol and handbook files live only under
`.ap/`:

```text
.ap/AP.md
.ap/AP_ORCHESTRATOR.md
.ap/AP_WORKER.md
.ap/PROMPT_CONTRACTS.md
.ap/ARTIFACT_LIFECYCLE.md
.ap/INTEGRATION.md
.ap/UPDATING.md
```

The root [AGENTS.md](../../AGENTS.md) remains a FrameNest-owned integration
document. Its managed AP block is generated by:

```text
./.ap/ap init
```

The managed block must remain canonical. FrameNest-specific rules live outside
that block. Universal AP rules must not be copied or customized in the
FrameNest root.

FrameNest removes the copied root AP handbooks and permanent BOOT/NEXT files.
Git history preserves the old content. Durable FrameNest facts from those files
must be migrated to the correct project-owned document before deletion. In this
migration, accepted command-observed NUC host facts were preserved in
[docs/NUC_HOST_BASELINE.md](../NUC_HOST_BASELINE.md).

Permanent repository BOOT and NEXT files are no longer the default lifecycle
model. Orchestrator rotation normally uses a generated professional restoration
prompt that contains the verified state needed by the fresh Orchestrator
instance. A restoration prompt grants no repository or host mutation authority.

A repository handoff is exceptional. It is appropriate only when material state
cannot be safely reconstructed from committed repository truth, public
verification, durable decisions, and the next authoritative task. If such a
handoff is needed, an Orchestrator issues a bounded Worker task that names the
exact path, consumer, lifecycle, allowed content, validation, Git authority, and
stop condition. The Worker writes and commits it under that authority.

Manual Cooperator `handout` commits are no longer the default AP rotation
mechanism for FrameNest because they put operational session state into the live
repository even when a generated restoration prompt and durable project records
are sufficient.

AP updates are explicit. A future update task must inspect the AP diff, move the
submodule to an exact reviewed commit, run AP candidate and strict doctor
checks, validate FrameNest compatibility, and commit the changed gitlink. The
FrameNest project must not consume AP `main` dynamically at runtime.

Rollback is a normal gitlink change: check out a known earlier AP commit inside
`.ap/`, run candidate validation, validate FrameNest, stage `.ap`, run strict
doctor, and commit the rollback gitlink.

## Clone And Initialization

Preferred clone:

```text
git clone --recurse-submodules <repository>
```

Recovery after an ordinary clone:

```text
git submodule update --init --recursive
./.ap/ap doctor
```

Participants must initialize the submodule before relying on `.ap/` protocol
files or AP tooling.

## Security And Supply Chain Consequences

Pinning AP by gitlink prevents silent protocol drift. FrameNest consumes the
exact AP commit recorded by the FrameNest commit, not whatever the AP remote
`main` points to later.

The integration tool validates the canonical AP repository identity,
`.gitmodules`, gitlink consistency, `.ap` cleanliness, and the exact managed
`AGENTS.md` block. It does not commit, push, copy AP protocol files, read
secrets, or modify global Git configuration.

The `.ap/` directory is read-only during ordinary FrameNest work. Modifying AP
requires a separate explicit AP update task. FrameNest-specific restrictions may
be stricter than AP, but they belong in FrameNest-owned documents rather than
local edits inside `.ap/`.

## Rejected Alternatives

### Copied Protocol Files

Rejected because root copies of universal AP files drift from the canonical
protocol, require manual replacement, and blur the boundary between universal
rules and project-specific rules.

### Customized Local AP Fork

Rejected because FrameNest does not need a project-local fork of universal AP.
FrameNest-specific constraints can be recorded in `AGENTS.md`, product docs,
security docs, deployment docs, or ADRs.

### Unpinned AP `main`

Rejected because dynamic consumption of remote `main` would silently change
project protocol behavior without a FrameNest review commit.

### Permanent NEXT Files

Rejected as the default because they turn session state into live repository
state and become stale. Repository handoffs remain available only as an
exceptional, explicitly authorized artifact.

### Permanent BOOT Files

Rejected because the canonical AP managed block plus `.ap/` protocol files and
project `AGENTS.md` provide the current bootstrap model without duplicating
universal handbooks.

### Manual Protocol Replacement

Rejected because manually replacing protocol files is error-prone and obscures
the exact version governing the project. A submodule gitlink is smaller,
reviewable, and reversible.

### Manual Cooperator Handoff Commits By Default

Rejected because routine handoff commits burden the Cooperator and preserve
temporary session state in the live tree. The default rotation artifact is now a
generated restoration prompt; committed handoffs are exceptional.

## Consequences

### Positive

- FrameNest has one AP integration model.
- Universal AP protocol files live in `.ap/`.
- FrameNest-specific policy remains in root `AGENTS.md`.
- AP updates and rollbacks become exact gitlink changes.
- Clone and recovery workflows use standard Git submodule commands.
- Stale BOOT/NEXT session state is removed from the live tree.
- Durable NUC host facts survive in a project-owned baseline document.

### Costs And Limitations

- New clones must initialize submodules or use recursive clone.
- Reviewers must inspect the `.ap` gitlink as part of AP-related changes.
- Generated restoration prompts are external to the repository unless an
  exceptional handoff artifact is explicitly authorized.
- Git history remains the archive for old protocol copies and session files.

## Verification Expectations

Repository verification should demonstrate:

- `.gitmodules` maps `.ap` to `https://github.com/cisarik/ap.git`;
- `.ap` records the accepted AP commit;
- `.ap` worktree is clean and detached at the recorded gitlink when expected;
- `./.ap/ap doctor` passes;
- root `AGENTS.md` contains the exact canonical managed AP block;
- no copied AP protocol files remain outside `.ap/`;
- no permanent BOOT/NEXT handoff files remain;
- README and development guidance document submodule clone and recovery;
- durable FrameNest rules and accepted NUC state were preserved;
- no secrets, host-specific identifiers, or private media names were migrated.

## Revisit Triggers

Revisit this decision if:

- AP adopts signed releases or another stronger distribution mechanism;
- Git submodules become unsuitable for FrameNest contributors or tooling;
- FrameNest needs an exceptional repository handoff that cannot be represented
  by a generated restoration prompt;
- the managed `AGENTS.md` block becomes insufficient for agent discovery;
- FrameNest needs stricter AP supply-chain verification than a pinned gitlink;
- AP changes its integration or update model materially.
