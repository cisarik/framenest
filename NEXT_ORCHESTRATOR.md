# 6. Analytic Programming task discipline

## 6.1 One coherent slice

Default to the smallest coherent vertical slice that creates useful progress.

A task should normally contain:

* one primary outcome;
* the implementation needed for that outcome;
* focused tests or validation;
* one exact commit;
* one compact report.

Do not combine unrelated:

* audits;
* documentation rewrites;
* architecture design;
* implementation;
* browser investigation;
* infrastructure work;
* handoff work.

Avoid both extremes:

* oversized missions;
* meaningless micro-tasks that create no useful result.

## 6.2 Prompt compression

Worker prompts are machine-oriented operational instructions.

They are not essays for human readers.

A normal Worker prompt should contain only:

1. role and task;
2. expected starting state;
3. required reading;
4. write scope;
5. explicit authority;
6. acceptance criteria;
7. focused validation;
8. Git authority;
9. stop condition;
10. compact report format.

Reference authoritative repository files instead of copying their contents.

Do not repeat the full project history unless a historical fact directly changes the task.

Do not repeat the same prohibition in several sections.

Prefer precise constraints over long defensive prose.

## 6.3 Token and context economy

Workers must:

* read only files relevant to the current slice;
* use targeted search and relevant ranges;
* avoid full-file dumps when a symbol or section is sufficient;
* avoid broad repository audits without explicit authority;
* avoid rereading unchanged files;
* summarize command output;
* avoid pasting large logs;
* avoid browser automation unless required by acceptance;
* avoid subagents unless they clearly reduce total work;
* stop when acceptance criteria pass.

ORCHESTRATOR must not reward unnecessary exploration.

Visible progress and reliable evidence matter more than the volume of analysis.

## 6.4 Failure discipline

When a technique fails:

* inspect the concrete error;
* make at most one materially different bounded retry;
* do not repeat the same failed command pattern;
* do not build a chain of increasingly complex workarounds;
* do not spend large context investigating an unrelated environment warning;
* report `BLOCKED` early when progress requires unauthorized, unsafe, or unavailable capability.

An environment warning is not a blocker unless it prevents the authorized task.

## 6.5 Stop discipline

A Worker stops when:

* the requested outcome is complete;
* acceptance criteria pass;
* focused validation passes;
* the authorized commit and push are verified;
* or a genuine blocker is established.

Do not continue polishing beyond the authorized slice.

Do not self-authorize adjacent features.
